## PHASE 1: Raw SelfGNN Baseline

This phase runs the unmodified SelfGNN on the binary bipartite graph. No edge weights, no node features. The code lives in `raw_selfgnn/`.

### Prompt 1.1 — Set up baseline codebase

```
Create the directory raw_selfgnn/ with 4 files: config.py, data_handler.py, model.py, train.py.

This is a PyTorch port of the original SelfGNN (Liu et al., SIGIR 2024).
The original TensorFlow code is provided in the conversation as reference documents.

Architecture (preserve exactly):
  1. Per-graph learnable embeddings: uEmbed[graphNum, user, latdim], iEmbed[graphNum, item, latdim]
  2. LightGCN message passing per time-interval graph:
     - Binary adjacency (no weights)
     - Symmetric normalization handled by sparse matrix
     - Residual connections: emb_new = propagated + emb_prev
     - Layer summation: final = sum of all layers (including layer 0)
  3. LSTM over time intervals (user and item separately)
  4. Multi-head self-attention over interval embeddings, then mean pool
  5. Instance-level sequence attention:
     - Look up item embeddings for user's sequence
     - Add positional embeddings
     - Stack of multi-head self-attention layers with LeakyReLU + residual
     - Sum pooling
  6. Prediction: dot(user_long, item_long) + dot(leaky_relu(seq_user), item_long)
  7. SAL loss: personalized weight network (3*d -> ssldim -> 1, sigmoid)
     aligns short-term and long-term pairwise preferences
  8. BPR margin loss: max(0, 1 - (pos_score - neg_score))

data_handler.py:
  - Loads trn_mat_time, sequence, tst_int, val_int, test_dict, val_dict
  - Builds binary sparse adjacency tensors for PyTorch
  - sample_train_batch: returns [pos_uids, neg_uids], [pos_iids, neg_iids], sequences, masks
  - sample_ssl_batch: samples pos/neg pairs per sub-graph for SAL
  - sample_eval_batch: supports mode='test' and mode='val'
  - Sequences are used as-is (no [:-1] slicing, test target is NOT appended)
  - test_dict and val_dict keys are 0-indexed

train.py:
  - Training loop with validation-based early stopping
  - Every tstEpoch epochs: evaluate on val set
  - Save best model by Val NDCG@10
  - After training: load best model, evaluate on test set
  - Save results to Results/config1_baseline.json with:
    { best_epoch, val_results, test_results, args, train_history }
  - train_history: list of {epoch, loss, val_HR10, val_NDCG10} per eval epoch

config.py defaults for Yelp:
  data=yelp-merchant, graphNum=5, gnn_layer=3, att_layer=2,
  batch=512, lr=1e-3, reg=1e-2, ssl_reg=1e-7, sslNum=40, sampNum=40,
  keepRate=0.5, leaky=0.5, latdim=64, ssldim=32, num_attention_heads=16,
  pos_length=200, pred_num=5, epoch=150, tstEpoch=3, patience=20,
  seed=100, device=cuda

Test that the code runs for 1 epoch without errors before full training.
```

### Prompt 2.2 — Build featured SelfGNN codebase

```
Create featured_selfgnn/ by copying raw_selfgnn/ and making these changes.

config.py — add three new flags:
  --use_edge_features (store_true, default False)
  --use_node_features (store_true, default False)
  --node_mlp_hidden (int, default 64) — hidden dim of the feature projection MLP

data_handler.py — add feature loading:

  In load_data(), after loading the standard files:

  If use_edge_features:
    - Load edge_weights.pkl
    - For each sub-graph adjacency matrix, replace binary values with
      edge weights where available (keep 1.0 where no weight exists)
    - Apply log-sigmoid: w_hat = sigmoid(log(1 + w))
    - Apply symmetric normalization: build D_row and D_col from w_hat,
      then w_normalized = w_hat[i,j] / sqrt(D_row[i]) / sqrt(D_col[j])
    - Build PyTorch sparse tensors from the weighted normalized matrix
    - When use_edge_features is False, build binary normalized adjacency
      (this is what raw_selfgnn does)

  If use_node_features:
    - Load user_features.npy and merchant_features.npy
    - Convert to torch tensors
    - Store as self.user_features and self.merchant_features

model.py — add feature integration:

  In __init__():
    If use_node_features:
      - Add self.user_mlp = nn.Sequential(
            nn.Linear(d_u, node_mlp_hidden),
            nn.ReLU(),
            nn.Linear(node_mlp_hidden, latdim))
      - Add self.merchant_mlp = nn.Sequential(
            nn.Linear(d_v, node_mlp_hidden),
            nn.ReLU(),
            nn.Linear(node_mlp_hidden, latdim))
      - Store the feature tensors as buffers

  In graph_encode():
    If use_node_features:
      - Compute f_u = self.user_mlp(user_features)  once per forward pass
      - Compute f_v = self.merchant_mlp(merchant_features)
      - At layer 0 of each sub-graph:
          u_embs[0] = self.user_embeds[k] + f_u   (additive fusion)
          i_embs[0] = self.item_embeds[k] + f_v

    The adjacency matrices (weighted or binary) are already handled by
    data_handler, so no change needed for edge features in model.py.

  Everything else stays identical to raw_selfgnn.

train.py — pass feature flags through, save config name that reflects
which features are enabled. Otherwise identical to raw_selfgnn/train.py.

After creating all files, test each configuration for 1 epoch:
  python train.py --data yelp-merchant --epoch 1  (baseline within featured code)
  python train.py --data yelp-merchant --epoch 1 --use_edge_features
  python train.py --data yelp-merchant --epoch 1 --use_node_features
  python train.py --data yelp-merchant --epoch 1 --use_edge_features --use_node_features

All 4 should complete without errors. Print the loss for each.
```

### Parallel execution opportunities

If you have multiple GPUs or want to run sequentially overnight, the most
efficient order is:

```
Day 1 morning:  0.1, 0.2, 1.1, 2.1
Day 1 afternoon: 1.2 (baseline training, ~2-3 hours)
Day 1 evening:  2.2 (build featured code + smoke test)
Day 1 overnight: 2.3 (edge training)

Day 2 afternoon: 2.5 (both training)
Day 2 evening:  3.1, 3.2, 3.3, 3.4, 3.5 (analysis + writing, ~1-2 hours)
```

### If a training run gives poor results

Before moving to Phase 3, check that baseline results are reasonable.
SelfGNN on original Yelp achieves HR@10 ≈ 0.365, NDCG@10 ≈ 0.201.
Your Yelp-Merchant dataset is different (different preprocessing, different
merchant set), so exact numbers will differ, but the order of magnitude
should be similar (HR@10 in 0.2-0.5 range).

If results are very low (HR@10 < 0.05), common issues:
  - sequence file has wrong format (empty sequences, wrong IDs)
  - test_dict keys are 1-indexed instead of 0-indexed
  - sub-graph adjacency matrices have timestamps instead of binary values
  - graphNum is larger than actual number of sub-graphs
